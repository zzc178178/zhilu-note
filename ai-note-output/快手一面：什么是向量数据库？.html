快手一面：什么是向量数据库？

📎 原文链接 https://mp.weixin.qq.com/s/9uHnv_h0IajG61KF6qQC1w
核心定义
向量数据库
专门用于存储和检索高维向量的数据库，核心能力是近似最近邻搜索（ANN），能在百万甚至亿级向量中快速找出最相似的几条。
本质差异
VS 传统数据库
MySQL 擅长精确匹配（WHERE id = 123），向量数据库擅长语义相似（「找和这个意思最接近的内容」），解决的是完全不同的问题。
典型场景
RAG 检索底座
用户问题 → 嵌入向量 → 向量数据库召回最相关文档片段 → 喂给大模型生成回答。RAG 中「检索」那一步的底层支撑。

📌 一、为什么需要专门的向量数据库？

普通关系型数据库（MySQL、PostgreSQL）靠 B-tree 索引，查询精确匹配效率极高。但向量检索不是找「等于」的，而是找「最相近」的——没有精确匹配，只有相似度排序。

高维向量（如 1024 维）如果暴力遍历，把查询向量和库里每一条都算一遍余弦相似度，百万条数据就要算一百万次，延迟完全不可接受。B-tree 只能处理一维有序索引，对高维向量「多个维度同时要考虑距离」的场景基本失效。

一句话总结： 向量数据库用专门的索引结构（HNSW、IVF）将 ANN 搜索加速到毫秒级，在可接受的精度损失下换取性能。

📌 二、核心索引算法

HNSW（Hierarchical Navigable Small World）

构建多层图结构，查询时从最上层稀疏图开始导航，逐层收窄范围，最终在底层找到最近邻。类似于在地图上找餐厅：不是遍历全国，而是先锁定方向，再从省→市→区一层一层缩小。

  • 优点 召回率高（95%+）、查询速度快
  • 缺点 建索引时内存消耗大
  • Qdrant、Milvus、Chroma 默认都用 HNSW

IVF（Inverted File Index）

先对向量做聚类，把相似的向量分进同一个「桶」，查询时只搜最相关的几个桶。类似于图书馆分类：找编程书，先找到「计算机科学」区域再搜，范围大幅缩小。

  • 优点 内存占用小，适合超大规模
  • 缺点 精度比 HNSW 略低，需要调参（nlist、nprobe）
  • Milvus 在超大规模场景下会用 IVF 系列索引

📌 三、生产级核心能力

除了 ANN 搜索，生产级向量数据库还需要支持：

  • Metadata 过滤（混合检索） — 给向量挂 metadata 字段，只在符合条件的子集里做 ANN，避免「先搜后滤」造成浪费
  • 实时更新 — 支持在线写入，新数据增量构建索引，无需停服重建
  • 与关键词检索融合 — 向量检索 + BM25 关键词检索混合召回，效果更好

📌 四、五款主流产品选型指南

Chroma — 开发测试首选

pip install 零配置上手，适合本地开发快速验证原型。但它是本地嵌入式数据库，不能用于生产环境。

Qdrant — 中小规模生产推荐

Rust 编写，性能稳定。API 简洁，Docker 一条命令部署。支持混合检索，很多团队原型用 Chroma，上生产就切到 Qdrant。

Milvus — 超大规模分布式

国内大厂（字节、小米）用得最多，支持多种索引类型，有完整集群方案，可水平扩展到亿级。但部署运维复杂度高。

Pinecone — 全托管云服务

SaaS 模式，按用量付费，无需运维，适合快速验证商业化产品。注意数据出境合规问题。

pgvector — PostgreSQL 插件

如果项目已有 PostgreSQL，直接用 pgvector 插件，不用引入新组件。查询可以和业务数据做 SQL JOIN，运维成本为零，但性能弱于专用向量库。

面试避坑提醒：
  1. 别把「MySQL 存向量字段」当向量数据库
  2. 别把 GitHub Star 数当唯一选型依据
  3. 应该说明各产品的适用场景和优劣势，体现选型思考深度
一句话选型： 开发用 Chroma → 中小生产用 Qdrant → 超大规模用 Milvus → 不想运维用 Pinecone → 已有 PG 用 pgvector
