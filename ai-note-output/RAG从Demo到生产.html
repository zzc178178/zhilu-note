RAG实战

RAG 从 Demo 到生产：四大关键环节完整拆解

鹅厂面试官追问：5000 份文档如何稳定答对？
原文来源 吴师兄学大模型 · 公众号

📌 核心观点

RAG 最难的不是跑通 Demo——Demo 只完成了 10% 的工作。真正让 RAG 在 5000 份文档中稳定答对，需要经历数据准备、检索召回、Query 理解、生成控制四大环节的系统性优化。

核心矛盾：Demo 阶段用几份文档跑通 pipeline 很容易，但扩展到生产环境后，chunk 碎片化、检索噪音、意图误解、LLM 幻觉等问题逐一暴露。

📦 数据准备：知识库质量决定 RAG 上限

常见错误

直接按固定字数切分 chunk（如 512 tokens），导致语义被切断，一个完整段落可能被拆到两个 chunk 中，检索结果碎片化。

优化方案

  • 结构化切分：按文档逻辑结构（标题、段落、表格、列表）而非固定字节切分
  • 上下文路径：每个 chunk 补充层级信息，如 产品名 > 章节 > 条款
  • 动态窗口：检索时返回命中 chunk 前后各一个 chunk，补充上下文
效果：召回准确率从 0.62 提升至 0.84

🔍 检索召回：不是找最相似的，是找最有用的

挑战

Embedding 模型选择、召回阈值设定、纯语义检索的关键词精确匹配缺陷。

解决方案

  • 中文场景使用 BGE-large-zh 替代 OpenAI 的 text-embedding-ada-002，语义理解更准确
  • 通过测试选定最佳召回阈值 0.72，兼顾召回率与精度
  • 采用 向量检索 + BM25 混合检索，再用 Reranker 模型 重新排序
效果：召回准确率从 0.84 提升至 0.91

🧠 Query 理解：用户问的，不一定是系统该搜的

问题

用户问题类型多样（事实查询、计算求解、NL2SQL、闲聊等），统一丢进向量检索会导致答非所问。

三级意图识别路由

  • 规则匹配：关键词/正则匹配，覆盖大部分常见问题，零延迟
  • BERT 分类器：模型分类，毫秒级响应，高准确率
  • LLM 兜底：处理疑难 case，最强理解但成本高

根据意图路由到不同模块：知识问答→混合检索 计算→计算模块 数据查询→NL2SQL 闲聊→通用对话

高级技巧

多索引路由：按主题分区索引，检索时只搜索相关分区，提高精度并降低延迟。

✍️ 生成阶段：RAG 不是简单的"检索+生成"

常见问题

LLM 自由发挥、模棱两可、答非所问。

优化方案

  • 强约束 Prompt：要求严格基于检索文档回答，禁止模糊表达，要求注明来源
  • 置信度评估：LLM 输出时附带 1-5 分置信度分数，低于 3 分自动触发二次检索或扩大检索范围
效果：答非所问投诉率从 18% 降到 7%

🔧 系统性工程协同

RAG 是端到端系统工程，每一细节都需要测试与调优：

  • 文档增量更新：知识库变更时的同步机制
  • 检索性能：Milvus 分区索引，只检索相关分区
  • 响应速度：高频 query 缓存、流式输出
  • 监控指标：召回准确率、投诉率、响应延迟持续跟踪

💡 面试回答框架

面试中聊 RAG 难点，建议按四个层次组织：

  1. 系统性思维 → RAG 是端到端工程，不止是"检索+生成"
