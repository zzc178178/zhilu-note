Agent 记忆提取触发策略：四大框架对比解析

Claude Code / Generative Agents / MemGPT / Mem0 触发机制深度解读
📎 原文链接 https://mp.weixin.qq.com/s/V3LBQYZBNGRku1vj2iB9bw
核心问题
记忆提取有代价
每次提取需额外调用 LLM API，有成本和延迟。需要在成本控制与记忆覆盖率间找平衡。
关键洞察
不是每轮都值得触发
用户说「好的」「明白」等无实质内容的对话，触发提取纯属浪费。
面试要点
从「每轮触发」到「智能策略」
能讲清楚四种策略的设计权衡，是面试中的关键分水岭。

📌 一、Claude Code：消息数量阈值 + Coalescing

触发条件：MIN_NEW_MESSAGES = 4

ExtractionCoordinator 在每轮 REPL 结束后被调用，但并非每次都真正触发提取：

increment = len(messages) - self._watermark
if increment >= MIN_NEW_MESSAGES:     # MIN_NEW_MESSAGES = 4
    await extract_memories(messages, memory_dir, model)
    self._watermark = len(messages)
为什么是 4 条？ 少于 4 条新消息说明刚开始对话或只是简单确认，提取出有意义信息的概率很低。4 条是「一次有实质内容对话交换」的最小粒度。

Coalescing：防止并发写乱

三个状态变量控制流程：

  • _running: bool — 是否有提取正在运行
  • _dirty: bool — 运行中是否有新请求进来
  • _watermark: int — 上次提取时的消息水位
关键保证： 中间请求不会丢失，会被「合并」到下一次提取。任意时刻最多只有一个提取任务在运行，避免并发写入。

📌 二、Generative Agents：重要性积分触发反思

Memory Stream + 重要性评分

所有经历的事件记录进 Memory Stream，每条记忆写入时 LLM 打重要性分数（1-10）：

  • 「我和 Isabella 聊了天」→ 6 分
  • 「我决定要在下周组织选举讨论」→ 9 分
  • 「我走过了图书馆」→ 1 分

触发逻辑

importance_sum += new_memory.importance_score
if importance_sum >= REFLECTION_THRESHOLD:  # 阈值 150
    run_reflection(recent_memories[-100:])
    importance_sum = 0
与 Claude Code 的本质区别： Claude Code 是「时钟驱动」（每 4 条消息扫一次），Generative Agents 是「事件驱动」（积累了足够重要的信息才触发）。

检索时的三维评分

score = α × recency_score + β × importance_score + γ × relevance_score
  • recency_score — 时间衰减，越近的记忆分越高（decay_rate ^ hours_passed，默认 0.995）
  • importance_score — 写入时打的重要性分
  • relevance_score — Query 与记忆的语义相似度

📌 三、MemGPT：LLM 自主决定何时读写记忆

虚拟内存类比

MemGPT 核心类比是操作系统的虚拟内存管理：

  • 主上下文（Main Context） = 当前正在用的内存，受 context window 大小限制
  • 外部存储（External Storage） = 磁盘上的页，可以无限大

LLM 通过函数调用主动管理存储

memory_search(query: str)           # 从外部存储检索记忆
memory_insert(content: str)         # 向外部存储写入记忆
core_memory_replace(key, new_val)   # 修改核心记忆
conversation_search(query: str)     # 搜索对话历史
设计代价： LLM 决策非确定性，可能忘记调用或在不必要时反复调用，消耗 token。记忆操作会与主对话耦合，增加实时响应延迟。

📌 四、Mem0：CRUD 四操作模型规范化提取

四操作分类

每次提取后，对已有记忆的处理只有四种操作：

操作 含义 示例
ADD 全新信息，之前没有 用户介绍了新项目
UPDATE 已有记忆的值发生变化 用户换了工作，更新职位
DELETE 已有记忆过时或被否定 用户说「我已经不用 mock 了」
NOOP 没有值得记的新信息 用户只说了「好的」
实测数据： 相比 OpenAI 方案提升 26% 准确率，降低 91% 的 p95 延迟，节省 90%+ token 成本。

📌 六、面试回答框架

「记忆提取有成本，每次触发要额外调一次 LLM，所以触发策略的核心是在成本控制和记忆覆盖率之间取平衡。」

两类主流策略：

  • 规则驱动：Claude Code 用消息数量阈值（≥4条），Generative Agents 用重要性积分阈值，触发时机确定可预测
  • LLM 自主驱动：MemGPT 让 LLM 通过函数调用自己决定，灵活但非确定性

提取结果处理： Mem0 用 ADD/UPDATE/DELETE/NOOP 四种操作，NOOP 表示没有有价值的新信息，避免无效写入。

项目实践话术：「在我们的 Agent 项目里，用的是类似 Claude Code 的后台异步策略，每 N 条新消息触发一次，用单独的任务异步执行不阻塞主对话。实测下来 API 调用次数比每轮触发降低了约 60%，记忆覆盖率没有明显下降。」

🎯 场景选型决策树

根据你的 Agent 类型选择合适的触发策略
  • 🤖 你的 Agent 需要记忆提取?
    • 需要实时响应用户 → 信息密度高? 社交/角色扮演
      • 高 → Generative Agents 重要性积分触发
      • 不高 → Claude Code 数量阈值+Coalescing
    • 不需要实时响应 → 需要长期记忆? 批处理/分析
      • 需要 → Mem0 CRUD 四操作模型 + NOOP 降本
      • 不需要 → 纯 context window 即可 无需复杂记忆
